SECTION 1 — CORE PRINCIPLES
====================================================================

 1. Database is the ONLY source of truth.
 2. FSM is NEVER source of truth — only a UI state guide.
 3. Every critical operation must be restart-safe.
 4. Every critical operation must be idempotent.
 5. Every critical operation must be audit-logged.
 6. Every critical operation must be recoverable.
 7. No critical state may depend solely on memory or FSM.
 8. All critical flows must be async-safe, duplicate-safe,
    and FloodWait-safe.
 9. Every user has exactly ONE permanent topic in the
    Verification Hub. That topic is the complete and permanent
    history of that user. It is never duplicated, never
    recreated unnecessarily, and never automatically deleted.
10. The user_id ↔ topic_id mapping stored in MongoDB is the
    canonical reference for all topic routing. All systems
    must reuse the existing topic.

SECTION 7 — PREMIUM SUBSCRIPTION SYSTEM
====================================================================

--- 7.1 PACKAGES ---

Packages:
  - 1 Month
  - 3 Months
  - 6 Months
  (Pricing configurable via config/ENV)

--- 7.2 SUBSCRIPTION MODEL ---

Statuses: ACTIVE | EXPIRED | CANCELLED | PENDING

Store (MongoDB — subscriptions collection):
  subscription_id
  user_id
  package_id
  started_at
  expires_at
  status

--- 7.3 FULL PAYMENT FLOW ---

User
  → Clicks Premium Access
  → Selects Package
  → Selects Payment Method
  → Payment Request Created (MongoDB record, PENDING)
  → Routed to User's Permanent Topic in Verification Hub
  → Admin Reviews Request in User Topic
  → Admin Manually Sends:
       • Number / QR / Wallet / Instructions
       (Admin types and sends directly in Telegram.
        Bot NEVER auto-sends payment details.)
  → Bot confirms Telegram delivery SUCCESS
  → Only AFTER delivery success:
       Payment Session Activated (DB)
       Session Timeout Timer Starts (20 minutes)
  → User Submits Payment Proof:
       • TXID (globally unique, validated)
       • Transaction Digits
       • Optional Screenshot
  → Bot Forwards Proof to User's Permanent Topic
  → Admins See in Topic:  ✅ Approve  |  ❌ Reject
  → Admin Approves → Subscription Activation Flow

CRITICAL RULES:
  • The bot must NEVER automatically send payment numbers.
  • The bot must NEVER automatically send wallet numbers.
  • Payment details are ALWAYS sent manually by admin.
  • Session starts ONLY after confirmed delivery success.
  • All payment activity routes into the user's permanent
    topic. No separate payment topic exists.

--- 7.4 PAYMENT SESSION RULES ---

  Timeout  : 20 Minutes (from delivery success, not request creation)
  Session  : DB-backed (MongoDB), never FSM-backed
  TXID     : Globally unique, duplicate auto-rejected
  Storage  : payment_sessions collection

--- 7.5 SUBSCRIPTION ACTIVATION (ATOMIC) ---

  System must in one atomic operation:
    1. Validate session is active
    2. Validate not already processed (idempotency key check)
    3. Acquire distributed lock (Redis)
    4. Activate subscription in MongoDB
    5. Generate unique one-time invite link (30 min expiry)
    6. Save invite metadata (invites collection)
    7. Save payment history (payment_history collection)
    8. Save invite history
    9. Clear session (payment_sessions collection)

--- 7.6 INVITE LINK RULES ---

  • One link per approval
  • One-time join only
  • 30-minute expiry
  • Expired → User contacts support → Admin generates new link

--- 7.7 SUBSCRIPTION LIFECYCLE NOTIFICATIONS ---

  Notify user:
    • 7 days before expiry
    • 3 days before expiry
    • On expiry day

--- 7.8 EXPIRY ENFORCEMENT ---

  • Expired user automatically removed from all premium groups
  • Group removal notification: ALLOWED
  • Bot mute/ban notifications: NOT SENT (silent)

SECTION 8 — TXID PROTECTION
====================================================================

  TXID must be globally unique across ALL records.

  Implementation:
    • Unique index on txid field in txid_registry collection
    • Check performed before any payment session acceptance
    • Race condition handled with distributed lock (Redis)

  Duplicate TXID:
    → Automatically rejected
    → Subscription activation blocked
    → Logged to Admin Logs topic
    → Logged to audit_logs collection

SECTION 9 — VERIFICATION HUB
====================================================================

Central Operations Center (Telegram Supergroup with Topics)

The Verification Hub is a single Telegram supergroup with
Forum (Topics) mode enabled. It serves as the command center
for all admin operations.

--- 9.1 HUB TOPIC STRUCTURE ---

  The hub contains exactly TWO categories of permanent topics:

  ┌─────────────────────────────────────────────────────┐
  │              VERIFICATION HUB                       │
  │                                                     │
  │  ├── 👤 User Topics  (one per registered user)      │
  │  └── 📋 Admin Logs   (one permanent topic)          │
  └─────────────────────────────────────────────────────┘

  REMOVED (no longer exist):
    ✗ Payments Topic       (replaced by User Topics)
    ✗ Support Topic        (replaced by User Topics)
    ✗ Content Review Topic (replaced by User Topics)
    ✗ User Moderation Topic(replaced by User Topics)
    ✗ Broadcast Topic      (removed)
    ✗ Audit Topic          (replaced by Admin Logs)

--- 9.2 USER TOPICS — ARCHITECTURE ---

  FUNDAMENTAL RULE:
    One User = One Permanent Topic.
    Forever. No exceptions.

  Topic naming format:
    👤 Full Name | User ID

  Examples:
    👤 Rahat Ferdous | 7535201490
    👤 Mayor | 2142417925

  Topic behavior:
    • Created once, on first admin-visible event for that user
    • Never duplicated
    • Never recreated unnecessarily
    • Never automatically deleted
    • Survives bot restarts
    • Survives admin changes
    • Is the complete permanent history of the user

  MongoDB source of truth:
    user_topics collection:
      user_id  (indexed, unique)
      topic_id (the Telegram forum topic ID)
      created_at

  All systems route into the user's existing topic.
  If a topic is somehow missing (topic was manually deleted
  by a Telegram admin), the bot must:
    1. Detect the error (Telegram API error on send)
    2. Create a new topic with the same name format
    3. Update the user_topics mapping in MongoDB
    4. Log the recovery event to Admin Logs
    5. Continue routing normally

--- 9.3 USER TOPIC PURPOSE ---

  Every user topic is the COMPLETE operational history of that user.
  ALL events for that user must be routed into that topic.

  Routed content includes (not exhaustive):

    Support
      • User support requests
      • Admin replies
      • Session open/close events

    Payments
      • Payment request card
      • Admin-sent payment details
      • User-submitted proof (TXID, digits, screenshot)
      • Approval or rejection notice
      • Subscription activation confirmation

    Subscriptions
      • Activation record
      • Expiry warnings
      • Renewal records

    Content Submissions
      • Submitted media card
      • Moderation buttons (Approve NSFW / Approve Premium / Reject)
      • Rejection reason
      • Approval confirmation

    Content Approvals & Rejections
      • Outcome of moderation per submission

    Warnings
      • Warning notice with reason

    Mutes
      • Mute notice with reason and duration

    Bans
      • Ban notice with reason

    Takedown Requests
      • Request card
      • Admin decision and reason

    Notes
      • Admin-written notes about user
      • Visible to all admins in topic

    Audit Events
      • Any system action targeting this user

--- 9.4 ADMIN LOGS TOPIC ---

  One permanent topic named:  📋 Admin Logs

  Receives a structured entry for EVERY admin action.

  Entry format:
    ┌─────────────────────────────────────────────┐
    │ [ACTION TYPE]                               │
    │ Admin     : [Admin Full Name]               │
    │ Admin ID  : [Admin User ID]                 │
    │ Target    : [Target Full Name] (@username)  │
    │ Target ID : [Target User ID]                │
    │ Detail    : [Action-specific detail]        │
    │ Time      : [ISO 8601 timestamp]            │
    └─────────────────────────────────────────────┘

  Action types that MUST produce an Admin Logs entry:

    PAYMENT APPROVED
    PAYMENT REJECTED
    PAYMENTDONE
    SUBSCRIPTION ACTIVATED
    SUBSCRIPTION EXPIRED
    SUPPORT ACCEPTED
    SUPPORT CLOSED
    CONTENT APPROVED NSFW
    CONTENT APPROVED PREMIUM
    CONTENT REJECTED
    USER WARNED
    USER MUTED
    USER UNMUTED
    USER BANNED
    USER UNBANNED
    USER KICKED
    NOTE ADDED
    TAKEDOWN APPROVED
    TAKEDOWN REJECTED
    INVITE GENERATED
    TXID DUPLICATE BLOCKED
    TOPIC RECOVERED

  Admin Logs topic_id is stored in MongoDB (hub_config collection)
  and must survive restarts.

--- 9.5 HUB ADMIN COMMANDS (inside Hub topics) ---

  Commands operable inside any User Topic:

    /accept      → Accept open support session for this user
    /close       → Close active support session
    /ban         → Ban this user (with reason prompt)
    /unban       → Remove ban
    /warn        → Issue warning (with reason prompt)
    /mute        → Mute user (with duration prompt)
    /unmute      → Remove mute
    /paymentdone → Mark payment manually done
    /profile     → Display full user profile card
    /history     → Display event history summary for user
    /note        → Add a note about this user
    /notes       → List all notes about this user

  Each command must:
    • Validate caller is OWNER or ADMIN
    • Execute the action
    • Post result to user's topic
    • Write entry to Admin Logs topic
    • Write entry to audit_logs collection

SECTION 10 — CONTENT SUBMISSION SYSTEM
====================================================================

--- 10.1 ENTRY REQUIREMENT ---

  Before submission is enabled, user must:
    1. Join Main Channel (verified via Telegram API)
    2. Accept Terms

--- 10.2 SUPPORTED CONTENT TYPES ---

  Photos | Videos | Albums | Documents | Audio |
  Voice  | GIF    | Files

--- 10.3 SUBMISSION FLOW ---

  User Submits Content (in bot chat)
    → Bot verifies Main Channel membership
    → Bot verifies terms accepted
    → Duplicate hash check (reject if duplicate)
    → Bot creates submission record (content_submissions collection)
    → Bot forwards/copies to User's Permanent Topic in Hub

  Admin sees in User Topic:
    ┌──────────────────────────────────────────────┐
    │ 📤 CONTENT SUBMISSION                        │
    │ From    : [Full Name] (@username)            │
    │ User ID : [id]                               │
    │ Type    : [Photo / Video / Album / ...]      │
    │ Hash    : [content_hash]                     │
    │ Time    : [timestamp]                        │
    └──────────────────────────────────────────────┘
    Buttons:
      [ 🔞 Approve NSFW ]  [ ⭐ Approve Premium ]  [ ❌ Reject ]

  NOTE: Anonymous moderation no longer exists.
  Admin always sees: Full Name, Username, User ID.

  On Approval (NSFW):
    → Content routed to NSFW Vault
    → Queue job created in NSFW Queue
    → User notified of approval
    → Admin Logs entry written
    → Audit log written

  On Approval (Premium):
    → Content routed to Premium Vault
    → Queue job created in Premium Queue
    → User notified of approval
    → Admin Logs entry written
    → Audit log written

  On Rejection:
    → Admin must provide rejection reason (mandatory, cannot skip)
    → Rejection reason saved to content_submissions record
    → User notified with rejection reason
    → Admin Logs entry written
    → Audit log written

--- 10.4 ALBUM HANDLING ---

  A Telegram media group (album) is treated as ONE logical item.

    • All media in the album are grouped by media_group_id
    • The album is received and buffered completely before
      forwarding to the user topic
    • The album is approved as a single unit
    • The album is stored as a single vault entry
    • The album is queued as a single queue job
    • The album is delivered as a single unit to target groups
    • Album hash = hash of all individual media hashes combined
      in order

  Albums must never be split.
  Albums must never be partially approved.

--- 10.5 CONTENT SAFETY ---

  - Duplicate detection via content hash (SHA-256 of media file_id)
  - Album hash fingerprinting
  - Fingerprint storage in content_fingerprints collection
  - Duplicate TXID-style rejection with audit log entry

SECTION 11 — VAULT SYSTEM
====================================================================

  Two strictly separated vault channels exist:

  ┌─────────────────────────────────────────┐
  │  🔞 NSFW Vault Channel                  │
  │  ⭐ Premium Vault Channel               │
  └─────────────────────────────────────────┘

  CRITICAL RULE:
    Content must NEVER cross between vaults.
    NSFW content stays in NSFW Vault exclusively.
    Premium content stays in Premium Vault exclusively.
    There is no shared or generic vault.

  Each vault stores:
    • Media (forwarded to vault channel)
    • Metadata (vault_items collection in MongoDB)
    • Approval data (approved_by, approved_at)
    • Hash for deduplication
    • Submitter user_id

  Each vault supports:
    • Random Reposting from vault when queue is empty
    • Historical Reuse
    • Fair Rotation (no item reposted too recently)
    • Cooldown Tracking (last_posted_at field)
    • Spam Prevention

  When fresh queue is empty:
    → Vault content distributed
    → Randomized selection with cooldown filter
    → Repetition prevention enforced
    → Fairness rotation enforced
    → NSFW Vault feeds NSFW distribution only
    → Premium Vault feeds Premium distribution only

  MongoDB — vault_items collection:
    vault_id
    user_id
    content_type       (photo / video / album / ...)
    vault_type         (nsfw / premium)
    telegram_message_id (in vault channel)
    media_hash
    approved_by
    approved_at
    last_posted_at
    post_count

SECTION 12 — QUEUE DISTRIBUTION ENGINE
====================================================================

--- 12.1 QUEUE STATES ---

  PENDING → PROCESSING → SENT
                       → FAILED → (retry) → DEAD_LETTER

--- 12.2 DUAL QUEUE ARCHITECTURE ---

  Two separate queues exist:

    🔞 NSFW Queue   — distributes to NSFW Group only
    ⭐ Premium Queue — distributes to Premium Group only

  Content must NEVER cross queues.
  Each queue is processed by its own independent worker.

--- 12.3 DEAD LETTER QUEUE ---

  Failed items exceeding retry limits → Dead Letter Queue
  Never retry forever.
  Dead letters require manual admin review.
  Dead letters stored in dead_letters collection.

--- 12.4 DISTRIBUTION PIPELINE ---

  Source Vault Channel
    → Ingestion
    → Queue Manager (NSFW or Premium, never mixed)
    → Vault Archive
    → Distribution Queue
    → Fairness Layer
    → Target Balancer
    → Rate Limiter
    → FloodWait Handler
    → Delivery Worker
    → Target Group (NSFW Group or Premium Group, respectively)

--- 12.5 DAILY LIMITS (ENV-CONTROLLED) ---

  PREMIUM_DAILY_LIMIT  (configurable via ENV)
  NSFW_DAILY_LIMIT     (configurable via ENV)

--- 12.6 QUEUE DB RECORD ---

  MongoDB — queue_jobs collection:
    job_id
    vault_id
    vault_type       (nsfw / premium)
    status           (PENDING / PROCESSING / SENT / FAILED / DEAD_LETTER)
    created_at
    scheduled_at
    processed_at
    retry_count
    error_message

SECTION 14 — TAKEDOWN REQUEST SYSTEM
====================================================================

--- 14.1 ENTRY ---

  User sends /takedown
  → Bot opens Takedown Request flow.

--- 14.2 REQUIRED INPUTS (collected step by step) ---

  Step 1: Bot asks → "What is your reason for removal?"
           User must type and send reason. (free text, mandatory)

  Step 2: Bot asks → "Send the link or reference to the content."
           User must send content link. (mandatory)

  Both fields are required. Neither can be skipped.
  If user sends /cancel at any step → flow cancelled silently.

--- 14.3 SUBMISSION ---

  After both inputs collected:
    → Bot confirms: "Your takedown request has been submitted."
    → Request saved to DB (PENDING) in takedown_requests collection
    → Request card posted to User's Permanent Topic in Hub

  Admin sees in User Topic:
    ┌─────────────────────────────────┐
    │ 🗑 TAKEDOWN REQUEST             │
    │ User   : [Name] (@username)     │
    │ User ID: [id]                   │
    │ Reason : [reason text]          │
    │ Link   : [content link]         │
    │ Time   : [timestamp]            │
    └─────────────────────────────────┘
    Buttons:
      [ ✅ Approve & Delete ]   [ ❌ Reject ]

--- 14.4 APPROVAL FLOW ---

  Admin clicks  ✅ Approve & Delete :

    Option A — Bot Auto-Delete:
      → Bot attempts to delete the content automatically.
      → On success: user notified "Your content removal request
        has been approved. The content has been removed."
      → Status updated to APPROVED in DB.
      → Admin Logs entry written.
      → Audit logged.

    Option B — Manual Delete (if bot cannot auto-delete):
      → Admin deletes the content manually.
      → Admin clicks "Confirm Manual Deletion" in topic.
      → Same user notification sent.
      → Status updated to APPROVED in DB.
      → Admin Logs entry written.
      → Audit logged.

--- 14.5 REJECTION FLOW ---

  Admin clicks  ❌ Reject :

    → Bot prompts admin in topic:
      "Type your rejection reason and send."

    → Admin must type and send reason. (mandatory, cannot be skipped)

    → User receives:
      "Your takedown request was reviewed and not approved.
       Reason: [admin reason]
       You may reply here to discuss this further."

    → Rejection reason saved to DB.
    → Status updated to REJECTED in DB.

    → A Support session is automatically flagged as open for
      this user. The user's permanent topic is already active —
      no new topic is created. The rejection reason is posted
      as a context note at the top of the support exchange
      in the user's existing topic.

    → Admin Logs entry written.
    → Audit logged.

--- 14.6 TAKEDOWN DB RECORD ---

  MongoDB — takedown_requests collection:
    takedown_id
    user_id
    reason
    content_link
    status              (PENDING / APPROVED / REJECTED)
    submitted_at
    reviewed_at
    reviewed_by
    rejection_reason

SECTION 15 — SUPPORT SYSTEM
====================================================================

  The support system is built on top of the permanent user topic
  architecture. There are NO per-ticket dedicated topics. All
  support history lives inside the user's permanent topic forever.

--- 15.1 ENTRY POINTS ---

  • Button: 🆘 Need Help
  • Command: /help
  • Any unhandled user text message sent to the bot
  • Auto-flagged on Takedown rejection (Section 14.5)

--- 15.2 SUPPORT SESSION IN USER TOPIC ---

  When a support event is triggered:
    → System checks user_topics collection for existing topic_id
    → If topic exists: post support request into existing topic
    → If topic missing: create topic, update mapping, post request

  Support request card posted to User Topic:
    ┌──────────────────────────────────────────────┐
    │ 🆘 SUPPORT REQUEST                           │
    │ From       : [Full Name] (@username)         │
    │ User ID    : [id]                            │
    │ Join Date  : [date]                          │
    │ Premium    : [Yes / No / Expired]            │
    │ Warnings   : [count]                         │
    │ Muted      : [Yes / No]                      │
    │ Banned     : [Yes / No]                      │
    │ Referrals  : [count]                         │
    │ Submissions: [total / approved / rejected]   │
    │ Message    : [user's message text]           │
    └──────────────────────────────────────────────┘
    Button: [ ✅ Accept Support ]

--- 15.3 ACCEPTANCE FLOW ---

  Support session status starts as PENDING.

  Admin sees: ✅ Accept Support button.

  On Accept:
    → Status updated to ACTIVE (support_sessions collection)
    → accepted_by = admin user_id
    → accepted_at = timestamp
    → Admin Logs entry written

  OWNERSHIP LOCK:
    Only ONE admin may hold an active support session at a time.
    If session is already ACTIVE:
      → Other admin sees: "Session already accepted by [Admin Name]"
      → Accept button is disabled/removed for others.

  5-Minute Unattended Rule:
    If PENDING session is not accepted within 5 minutes:
      → User receives: "No admin available currently.
        Your request has been noted. An admin will respond
        when available."
      → Session status remains PENDING (admin may still accept)
      → This notification fires only once per session

--- 15.4 TWO-WAY CONVERSATION BRIDGE ---

  Admin Reply in Topic → Bot forwards to User
  User Reply in Bot    → Bot forwards to User Topic

  Full two-way live bridge.
  Bridge is active while session status = ACTIVE.
  Bridge is keyed on user_id → topic_id mapping.

--- 15.5 CLOSURE ---

  Admin issues /close in User Topic:
    → Status updated to CLOSED
    → closed_by = admin user_id
    → closed_at = timestamp
    → Closure summary posted in User Topic
    → User-side conversation: user notified "Support session closed."
    → User-side messages for this session: DELETED
    → Admin Logs entry written
    → Audit logged

  Topic in Hub: PRESERVED permanently
  Logs: PRESERVED permanently
  Closure summary stored in support_sessions collection.

  After closure: bridge is deactivated for this session.
  A new /help starts a new session card in the same topic.

--- 15.6 SUPPORT DB RECORD ---

  MongoDB — support_sessions collection:
    session_id
    user_id
    topic_id         (always = user's permanent topic_id)
    status           (PENDING / ACTIVE / CLOSED)
    opened_at
    accepted_by
    accepted_at
    closed_by
    closed_at
    trigger          (manual / takedown_rejection / unhandled_message)

SECTION 25 — RESTART SAFETY (DB STORAGE MAP)
====================================================================

  CRITICAL RULE: The following must NEVER live in FSM only.
  Every piece of state must be written to MongoDB FIRST,
  before any Telegram action is taken. On restart, the bot
  reads MongoDB and restores all active state.

  State that must survive restart:

    user_topics
      → user_id ↔ topic_id mappings
      → Must be reloaded into memory cache on startup
      → Used by every routing operation

    support_sessions
      → All PENDING and ACTIVE sessions restored
      → Bridge re-established for ACTIVE sessions
      → Ownership lock re-applied (accepted_by field)

    payment_sessions
      → All active (non-expired) sessions restored
      → Timeout timers re-registered from (activated_at + 20min)
      → Sessions past timeout are expired on restart

    queue_jobs
      → All PENDING and PROCESSING jobs restored
      → PROCESSING jobs reset to PENDING (were interrupted)
      → Workers re-queued and resume

    subscriptions
      → All ACTIVE subscriptions restored
      → Expiry timers re-registered from expires_at
      → Any subscription expired during downtime is processed
        on restart

    referrals
      → Fully persisted. No in-memory state required.
      → Loaded on demand.

    moderation actions (punishments)
      → All active bans, mutes stored in MongoDB
      → Restored and enforced on restart

    takedown_requests
      → All PENDING requests restored for admin review
      → No re-routing needed (already in user topics)

    hub_config
      → Admin Logs topic_id
      → Main Channel ID
      → NSFW Group ID
      → Premium Group ID
      → NSFW Vault Channel ID
      → Premium Vault Channel ID
      → Loaded on startup, cached
