## Senior QC / Production Sign-Off Review

## Executive Summary

After a full read of all 124 source files and the cross-check report, this audit confirms **8 critical systems were either missing, incomplete, or broken**. This document catalogs every gap found and the exact files that implement the fixes.

The existing codebase (Pyrogram + Motor + MongoDB) is architecturally sound. The core engine (queue, watermark, distribution, vault) is well-built. The gaps are concentrated in user-facing flows and lifecycle systems.

## Critical Gaps Found & Fixed

### 🔴 GAP 1 — Payment System (System 5): INCOMPLETE

**What existed:** `app/payments/handlers.py`, `app/payments/service.py`, `app/payments/models.py`, `app/payments/repository.py`

**What was broken:**
- `approve_payment()` called `ParseMode.HTML` without importing it
- `resume_active_sessions()` called `start_timeout()` but the session already had `expires_at` set — it would reset the timer on restart instead of using remaining time
- `PaymentStatus.PENDING_DETAILS` transition from `REQUESTED` existed but `handle_payment_method` tried to transition directly to `REQUESTED` which isn't an allowed transition from `WAITING_PAYMENT_DETAILS`
- The admin "Send Payment Details" flow routed through `pay:admin:send:` callback but `handle_payment_inputs` used group -10 priority which conflicts with support handler at default priority
- Session creation: `offset = round(random.uniform(0.01, 0.50), 2)` — `random` not imported in service
- `handle_admin_decision` called `callback.message.edit_caption` but payment card was sent as text not photo for initial request — would throw `MessageNotModified` or wrong method
- `_notify_admins_of_request` uses `PaymentSession` type hint but the object is referenced before type is imported

**Fix:** `app/payments/service.py` — rewritten with all imports, correct FSM transitions, restart-safe timer resume using remaining time from `expires_at`, full atomic approval.

---

### 🔴 GAP 2 — Subscription Lifecycle (System 6): MISSING SCHEDULER REGISTRATION

**What existed:** `app/workers/subscription_worker.py` — well-implemented

**What was broken:**
- `SubscriptionWorker` was started in `lifecycle.py` BUT the 7-day / 3-day notification logic existed but used `metadata` dict to track whether warnings were sent — this dict is on the Python object only, not persisted to DB. On restart, ALL warnings would resend.
- `seven_day_warned`, `three_day_warned_1`, `three_day_warned_2` were set on `sub.metadata` and then `update_subscription(sub)` was called — but `update_subscription` just calls `repo.upsert(sub)` which overwrites the entire subscription document including metadata. This works IF the subscription model's `to_dict()` includes metadata. It does. ✅
- However: `three_day_warned_2` logic checked `expires_at < now + timedelta(hours=48)` but `get_expiring_soon(within_hours=72)` returns subs expiring within 72h — meaning the second warning fires immediately after the first for subs with 48-72h remaining, not 24h after first.

**Fix:** `app/workers/subscription_worker.py` — timing fix using timestamp-based check instead of window overlap.

### 🔴 GAP 3 — TXID Protection (System 7/8): PARTIALLY IMPLEMENTED

**What existed:** `PaymentRepository` has `pay_txid_unique` index (sparse, unique on `txid`). `handle_payment_inputs` catches `DuplicateKeyError`.

**What was broken:**
- The `DuplicateKeyError` import was missing from `app/payments/handlers.py` — the catch block would itself throw `NameError`
- The duplicate TXID path called `service.reject_payment(session.id, ..., 0)` but `reject_payment` calls `acquire_processing_lock` first, and the session status is `AWAITING_PAYMENT` not `UNDER_REVIEW` — lock acquisition would fail (returns False) and the session would not be rejected
- User was banned but session was left in `AWAITING_PAYMENT` state — would allow retry attempts

**Fix:** `app/payments/handlers.py` — added `DuplicateKeyError` import, fixed reject path to force-update status directly.

### 🔴 GAP 4 — Broadcast Missing Content Types (System 9): PARTIALLY IMPLEMENTED

**What existed:** `app/handlers/admin_handler.py` has broadcast system handling text, photo, albums, documents.

**What was broken:**
- `_safe_send_broadcast` used `message.copy(chat_id=user_id)` — Pyrogram's `Message.copy()` is not a method. Should be `client.copy_message(chat_id, from_chat_id, message_id)`
- Video, voice, audio, animation, sticker not explicitly handled — `copy_message` would handle them but the media_group collection had no timeout window
- Caption was NOT preserved — `copy_message` without `caption` parameter drops captions for non-album messages
- `broadcast_data["content"]` stored the `Message` object but `_execute_broadcast` was a separate task — if bot restarted mid-broadcast, it would be lost (FSM-only, not DB-backed)

**Fix:** `app/handlers/admin_handler.py` — fixed `_safe_send_broadcast`, added media_group collection, caption preservation, all content types.

### 🔴 GAP 5 — Content Submission (System 12): INCOMPLETE

**What existed:** `app/handlers/submission_handler.py` — exists but incomplete.

**What was broken:**
- Channel membership gate (`VAULT_CHANNEL_ID`) was NOT implemented — the spec requires user to join the main channel before submitting
- Terms acceptance stored in `consent_records` with `is_active=True` — but `submission_handler.py`'s `handle_consent_agree` callback (`consent:agree` without user ID) conflicted with `creator_onboarding.py`'s `consent:agree:{user_id}` pattern
- `archive_to_vault` was called with `dest="pending"` which is not a valid `ModerationDestination` — would cause issues in vault queries
- `_finalize_submission` called `register_pending` AFTER `forward_to_verification` — but `pop_pending` in callback handler looks up by `first_msg_id` from the ORIGINAL message, not the forwarded message. The pending registry should be keyed by the original message IDs.
- `WeakValueDictionary` for album locks — if no strong reference is held to the lock between `create_task` and `async with lock`, the lock gets GC'd

**Fix:** `app/handlers/submission_handler.py` — fixed membership gate, consent flow, vault dest, pending registry key, album lock lifecycle.

### 🔴 GAP 6 — Vault & Queue Engine (System 13): MOSTLY WORKING, GAPS

**What existed:** `app/scheduler/scheduler.py`, `app/repositories/queue_repository.py`, `app/distribution/engine.py` — substantial and well-built.

**What was broken:**
- `_enqueue_content` in scheduler used `settings.VAULT_CHANNEL_ID` for `vault_chat_id` — correct — but `vault_message_id` was `content_item.get("vault_message_id")` which could be `None` for legacy vault items. The queue repo's `enqueue()` throws `VaultReferenceMissingError` if `vault_message_id` is None. The scheduler would silently drop these items.
- Dead letter queue existed in repo but `move_to_dead_letter` updated the original job to `DEAD` status but the `dead_letters` collection `upsert` used `original_job_id` as unique key — if called twice (retry race), would silently fail second upsert
- `get_deadline_exceeded_jobs` queried `scheduled_at` but the model field is `queue_deadline` — no results ever returned
- Daily cap check in `_distribution_cycle` used `_get_posted_count_last_24h` which queries `completed_at >= cutoff` — correct. But `_get_daily_cap` used `dest` from `source_channel_id.replace("submission_", "")` which only works for `submission_nsfw` / `submission_premium` — direct vault uploads (from `backfill_vault.py` or `vault_handler.py`) use `content_item.get("content_id")` not `submission_` prefix

**Fix:** Scheduler and queue repo — fixed field names, vault_message_id fallback, deadline query field, daily cap dest resolution.

### 🔴 GAP 7 — Message Cleanup (System 17): 4 OF 5 RULES MISSING

**What existed:** `app/services/cleanup_service.py` — partial implementation. `app/handlers/update_logger.py` — phone number detection exists but only logs, doesn't schedule deletion. `app/core/lifecycle.py` registers cleanup sweep every 5 minutes.

**What was broken:**
- `CleanupService.log_message` correctly categorizes messages but the `run_cleanup_sweep` only deletes via `self._bot.delete_messages(user_id, msg_id)` — this is a DM delete, won't work for group messages
- BD phone number regex in `cleanup_service.py` is `(?:017|018|019|014|013)\d{8}` (9 total digits without leading 0, making 11-digit numbers) — correct
- BUT `update_logger.py` checked `if any(p in text for p in ["017", "018", "019", "014", "013"])` — this would match ANY text containing those strings (e.g. "2017 was a good year")
- Payment session message cleanup: no payment handler called `log_message` with `category="payment"` — the 20-minute timer only ran for explicitly logged messages
- Support closure cleanup: `delete_user_support_history` existed but `support_handler.py` never called it in the closure callback — it was only called from `handle_support_closure_callback` which had a bug: `tid` variable referenced in audit log but came from the regex match group named `tid` which was the ticket_id string, not user_id

**Fix:** `app/services/cleanup_service.py`, `app/handlers/support_handler.py`, `app/handlers/update_logger.py` — complete the 5 cleanup rules.

### 🔴 GAP 8 — Restart Safety (System 20/25): PARTIALLY IMPLEMENTED

**What existed:** `lifecycle.py` calls `payment_service.resume_active_sessions()` as a task.

**What was broken:**
- `resume_active_sessions` called `start_timeout(session.id)` unconditionally — this RESET the timer to `now + 20 minutes` instead of restoring the remaining time from `session.expires_at`
- Support tickets: no restoration of in-memory topic cache on restart — `TopicManager._local_cache` is empty on restart, triggering new topic creation for existing tickets
- Queue: `recover_stale_jobs()` was called at startup (in `DispatcherWorker.start()` and scheduler's `_run_startup_integrity_scan`) — this IS implemented correctly ✅
- Payment FSM: `get_redis()` returns a sync-style client but `handle_payment_inputs` used `await redis.exists(...)` — mixing sync/async Redis

**Fix:** `app/payments/service.py`, `app/services/topic_manager.py` — restart-safe timer resume, topic cache restoration.

## Additional Bugs Found (Not in Cross-Check)

### BUG A — `app/handlers/callback_handler.py`: Missing import
`get_redis` used but not imported in `handle_moderation_callback` and `handle_mod_reject_reason_reply`.

### BUG B — `app/handlers/admin_handler.py`: Duplicate handler definitions
`handle_ban_command` and `handle_unban_command` are each defined TWICE with different `@permission_required` decorators. Pyrogram will register both; the second one overwrites the first silently.

### BUG C — `app/services/audit_service.py`: Missing imports
`settings` used but not imported. `ParseMode` used but not imported.

### BUG D — `app/handlers/topic_router.py`: `ChatType` used but not imported
`chat.type == ChatType.FORUM` in `topic_manager.py` references `ChatType` imported from `pyrogram.enums` but the import line is missing.

### BUG E — `app/referral/service.py`: `ReturnDocument` not imported
`from pymongo import ReturnDocument` missing — `reward_approved_content` would crash.

### BUG F — `app/payments/service.py`: `ParseMode` not imported in approve flow
`await client.send_message(..., parse_mode=ParseMode.HTML)` — no import.

### BUG G — `app/watermark/ffmpeg_processor.py`: `settings.WATERMARK_POSITION` not defined
The `_get_watermark_config` function in `moderation_actions.py` sets `position: settings.WATERMARK_POSITION` but `Settings` class doesn't have this field — only `WATERMARK_OPACITY`, `WATERMARK_SCALE`, etc. The field was missing from `settings.py`.

**Wait** — checking `settings.py`... `WATERMARK_POSITION` IS missing. The `WatermarkPosition` enum default in `ffmpeg_processor.py` hardcodes `BOTTOM_RIGHT`. But `moderation_actions.py` passes `settings.WATERMARK_POSITION` to watermark_config dict, which would fail at settings load with `AttributeError`.

**Fix:** `app/config/settings.py` — add `WATERMARK_POSITION: str = "BOTTOM_RIGHT"`.

## Architecture Notes for Production

**Scaling:** The current architecture supports horizontal scaling of dispatcher and watermark workers via `docker-compose up --scale dispatcher=3`. The distributed lock service prevents duplicate delivery.

**MongoDB:** Without replica set, transactions fall back to non-atomic operations. The code handles this gracefully but TXID protection and album delivery guarantees are weaker. Use Atlas or a self-hosted replica set.

**Session persistence:** All critical state (payments, support, queue, invites) is DB-backed. A full process restart loses only in-memory rate limiter bucket state and flood wait timers (which are also Redis-backed now).

**FFmpeg:** The Dockerfile installs FFmpeg. For Railway deployment, the `nixpacks` config in `railway.toml` also installs it. Video watermarking requires ~2-4x video duration in processing time at `ultrafast` preset.
