## AUDIT CHECKLIST

For each file you read, check and report on every item below.
Mark each finding as one of:
  ✅ PASS       — confirmed working, no action needed
  ⚠️ WARNING    — works but has risk or tech debt
  ❌ FATAL      — will crash or silently corrupt data in production
  🔍 UNVERIFIED — could not confirm without runtime data

### 1. BOOT SEQUENCE
- Does main_bot.py complete without crashing?
- Does lifecycle.py start Pyrogram successfully?
- Is every startup step reachable (not inside dead code blocks)?
- Does MongoDB connect before any repository is used?
- Are indexes created without duplication errors?
- Does the scheduler start and register jobs correctly?
- Are all workers instantiated and started?
- Does the health server bind correctly?

---

### 2. LOGGER USAGE (CRITICAL)
Scan every single logger call in every file.
The logger is stdlib logging.Logger.
The ONLY valid calling conventions are:
  logger.info("message")
  logger.info("message", extra={"key": value})
  logger.error("message", exc_info=True)

Flag every instance of:
  logger.info("msg", key=value)      ← TypeError, crashes process
  logger.info("msg", stage=value)    ← TypeError, crashes process
  logger.debug("msg", ctx=value)     ← TypeError, crashes process

Report file name, line number, and exact bad call for every violation.

---

### 3. IMPORT INTEGRITY
- Are there any missing imports (NameError on first use)?
- Are there any circular imports?
- Are there any imports of undefined symbols from internal modules?
- List every broken import with file and line.

---

### 4. DEAD CODE & UNREACHABLE BLOCKS
- Is any critical code placed after sys.exit() or inside an except block that exits first?
- Are any startup steps silently skipped?
- Are there any methods called that do not exist on the target class/object?
- List every unreachable block that contains production logic.

---

### 5. REPOSITORY CONTRACT VERIFICATION
For every method called on a repository class anywhere in services, workers, or scheduler:
- Confirm the method actually exists on that repository class.
- Confirm the method signature matches the call site.
- Flag every call to a non-existent repository method.

---

### 6. WORKER & SCHEDULER INTEGRITY
- Is every worker instantiated in lifecycle.py?
- Is every worker started in lifecycle.py?
- Is every worker stopped gracefully on shutdown?
- Does the scheduler have all required imports?
- Does setup_jobs() or equivalent exist and get called?
- Is stale singleton lock cleaned on restart?
- Are all APScheduler jobs pointing to existing async methods?

---

### 7. QUEUE & VAULT SAFETY
- Does every enqueue path validate vault_chat_id + vault_message_id before inserting?
- Are albums processed as atomic units?
- Is there a dead-letter path for max-retry-exceeded jobs?
- Is content always delivered via copy_message() / copy_media_group()?
- Is forward_messages() used anywhere? (FORBIDDEN — leaks identity)
- Are all queue state transitions atomic ($set / find_one_and_update)?

---

### 8. TELEGRAM DELIVERY SAFETY
- Is FloodWait caught and respected on every Telegram API call?
- Is there retry logic with backoff on every delivery path?
- Are flood wait durations persisted (Redis or DB) so workers don't retry too early?
- Is there any place where a failed Telegram call silently drops a job?

---

### 9. STATE CONSISTENCY
- Are pending submissions kept in sync between in-memory dict and MongoDB PENDING_COLLECTION?
- Are payment sessions only activated after confirmed Telegram delivery?
- Are payment timeouts only started after session activation?
- Is TXID globally unique and checked before approval?
- Are duplicate processing scenarios prevented with locks?

---

### 10. REDIS USAGE
- Is the Redis client correctly named and imported everywhere it is used?
- Is there any NameError risk on Redis access (_redis vs redis vs client)?
- Are all Redis operations wrapped with proper error handling?

---

### 11. MONGODB SAFETY
- Are any collections indexed twice in _ensure_indexes() (causes DEGRADED state)?
- Are all hot-path queries using indexed fields?
- Is there any direct collection access from handlers or lifecycle.py?
- Are all migrations idempotent?

---

### 12. ASYNC SAFETY
- Is there any blocking I/O (file read, subprocess, sleep) in async handlers or workers?
- Is asyncio.create_subprocess_exec() used instead of subprocess.run() in async paths?
- Are all background tasks tracked and cancellable?
- Is there any fire-and-forget task without error handling?

---

### 13. EXCEPTION HANDLING
- Is there any bare `except:` without logging?
- Are exceptions swallowed silently anywhere?
- Is every critical failure logged with exc_info=True?
- Are non-fatal errors isolated so they don't crash the entire process?

---

### 14. GRACEFUL SHUTDOWN
- Does shutdown drain in-flight queue jobs?
- Does shutdown cancel all background tasks cleanly?
- Does shutdown close MongoDB connection?
- Does shutdown stop the Pyrogram client cleanly?
- Is there any resource that leaks on restart?

---

### 15. HANDLER LAYER CLEANLINESS
- Do handlers contain any business logic? (They must not — delegates only)
- Do handlers call repositories directly? (Forbidden — must go through services)
- Do handlers call get_db() directly? (Forbidden)
- Are all handlers thin: validate → delegate → respond?

---

### 16. CONFIGURATION & ENV SAFETY
- Are all required ENV variables validated on startup?
- Does the system fail fast with a clear error if a required variable is missing?
- Are secrets (tokens, passwords) never logged?
