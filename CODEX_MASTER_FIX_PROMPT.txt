STEP 5 — CRITICAL RULES TO ENFORCE IN EVERY FILE
====================================================================

For each file you touch, enforce ALL of the following:

  IMPORTS
    ✓ Pyrogram only for Telegram (not aiogram, not telethon)
    ✓ motor.motor_asyncio for MongoDB
    ✓ aioredis for Redis
    ✓ All local imports use correct paths matching actual file structure

  ASYNC
    ✓ Every function that touches DB or Telegram is async def
    ✓ Every Motor call has await
    ✓ Every Pyrogram call has await
    ✓ No blocking calls inside async functions (no time.sleep,
      use asyncio.sleep instead)

  DATABASE WRITES
    ✓ MongoDB write happens BEFORE the Telegram message is sent
    ✓ All collection names match Section 25A of the spec exactly:
        users, user_topics, subscriptions, payment_sessions,
        payment_history, txid_registry, invites, support_sessions,
        content_submissions, content_fingerprints, vault_items,
        queue_jobs, dead_letters, referrals, punishments,
        takedown_requests, audit_logs, admins, hub_config
    ✓ No hardcoded collection name strings — use constants

  ATOMIC OPERATIONS
    ✓ Any operation that activates subscriptions, generates invites,
      or processes payments must acquire a Redis distributed lock first
    ✓ Lock acquired → DB write → Telegram send → Lock released
    ✓ Idempotency key checked before subscription activation

  ERROR HANDLING
    ✓ Every async function has try/except
    ✓ FloodWait caught: except FloodWait as e: await asyncio.sleep(e.value)
    ✓ No bare except: pass
    ✓ All exceptions logged with logger.exception() or logger.error()
    ✓ Errors written to audit_logs collection when they affect users

  ROUTING
    ✓ All user routing goes through user_topics collection lookup
    ✓ user_id → topic_id is always fetched from MongoDB
    ✓ If topic_id not found: create topic, save to user_topics, use it
    ✓ If Telegram returns error on topic send: recover (see spec 9.2)

  ADMIN ACTIONS
    ✓ Every admin action posts structured entry to Admin Logs topic
    ✓ Every admin action writes to audit_logs collection
    ✓ Both writes happen even if one fails (independent try/except)

  ROLES
    ✓ Only two roles exist: owner and admin
    ✓ All role checks use only these two values
    ✓ Remove or neutralize any MODERATOR, SUDO, SUPER_ADMIN,
      SUPPORT_ADMIN, PAYMENT_ADMIN, SCHEDULER_ADMIN checks

  CONTENT SEPARATION
    ✓ NSFW content goes to NSFW vault → NSFW queue → NSFW group only
    ✓ Premium content goes to Premium vault → Premium queue → Premium group only
    ✓ No content can cross between NSFW and Premium systems

  PAYMENT RULES
    ✓ Bot NEVER auto-sends payment numbers
    ✓ Bot NEVER auto-sends wallet numbers
    ✓ Admin manually sends payment details in topic
    ✓ 20-minute timeout starts AFTER delivery confirmation, not on request

  RESTART SAFETY
    ✓ No critical state lives only in FSM
    ✓ All session state is in MongoDB
    ✓ On startup: restore all PENDING/ACTIVE sessions from MongoDB
