## BUG 4 — Structlog API misuse: unexpected keyword 'error_type'
FILES: app/core/lifecycle.py AND main_bot.py

SYMPTOM:
  TypeError: Logger._log() got an unexpected keyword argument 'error_type'
  at: logger.exception(..., error_type='...')

ROOT CAUSE:
  The code is using stdlib logging.Logger.exception() which does not accept extra keyword arguments beyond 'exc_info', 'stack_info', 'stacklevel'.
  Structlog bound loggers DO accept arbitrary keyword args.
  Either: (a) the logger is a stdlib logger being used with structlog syntax, OR (b) structlog is not properly configured.

FIX:
  Read app/core/lifecycle.py and main_bot.py first.

  Option A (preferred): If the file already imports structlog and uses structlog.get_logger(), the issue is that somewhere in the logging chain it falls back to stdlib. Fix the structlog configuration to use structlog.stdlib.ProcessorFormatter properly.

  Option B: If logger = logging.getLogger(__name__) is used anywhere, replace with:
    import structlog
    logger = structlog.get_logger(__name__)

  For structlog loggers, the correct call is:
    logger.exception('message', error_type='startup_failure', exc_info=True)
  OR:
    logger.error('message', error_type='startup_failure', exc_info=True)

  For stdlib loggers, extra kwargs go in the 'extra' dict:
    logger.exception('message', extra={'error_type': 'startup_failure'})

  Find every occurrence of logger.exception(..., error_type=...) and logger.error(..., error_type=...) in both files and fix them to match the actual logger type being used.
